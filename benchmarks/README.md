# wizard performance benchmarks

This is the **offline** performance bench harness. It validates the
migration plan's performance claims that the wizard's existing tests
don't cover — prompt-cache hit rate, system-prefix token reduction,
first-token latency, bundle-size delta, and per-tool execution time.

> Not to be confused with `src/lib/middleware/benchmarks/` — those are
> **runtime per-turn telemetry trackers** that capture observability
> data during real wizard runs. This directory is a separate, offline
> harness invoked by `pnpm bench`.

## What each bench measures

| Bench | What it measures | How |
| --- | --- | --- |
| `prefix-size` | System-prefix tokens with every skill body inlined ("eager") vs. with a skill menu only ("lazy"). | Walks `skills/` + `.claude/skills/`, sums skill-body bytes, compares against a menu-only construction over `getWizardCommandments({})`. |
| `bundle-size` | npm tarball bytes for the current repo, optionally compared against a baseline checkout. | `pnpm pack --pack-destination <tmp>` on each repo; stat the `.tgz`. |
| `tool-exec-time` | Per-tool wall-clock cost: `read_file`, `list_files`, `grep`, `edit_apply`, `load_skill`. | n=100 iterations on a fixture project; reports min / median / p95 / max. |
| `cache-hits` *(live)* | Anthropic prompt-cache hit rate across two consecutive turns. | Requires `WIZARD_LIVE_BENCHMARK=1`. Currently stubbed — see TODO below. |
| `first-token-latency` *(live)* | First text-delta latency, cold vs. warm. | Requires `WIZARD_LIVE_BENCHMARK=1`. Currently stubbed — see TODO below. |

## Running

```sh
pnpm bench           # offline only (default)
pnpm bench --live    # also runs cache-hits and first-token-latency
                     # (still requires WIZARD_LIVE_BENCHMARK=1 + OAuth)
pnpm bench --json    # emit results.json contents on stdout instead of
                     # the markdown table
```

Output:

- Console: a markdown table.
- `benchmarks/results.json`: machine-readable run record, overwritten
  on every run and **gitignored** — CI uploads it as an artifact via
  `actions/upload-artifact` (see `.github/workflows/bench.yml`).
- `benchmarks/baseline.json`: snapshot for trend tracking. Refresh after
  a known-good run with `cp benchmarks/results.json benchmarks/baseline.json`.

## How to interpret `baseline.json` vs `results.json`

- **`results.json`** is regenerated by every `pnpm bench` invocation.
  It reflects the numbers from *this run* only.
- **`baseline.json`** is the reference snapshot. The committed file
  starts as a **placeholder with zero values and `skipped` statuses** —
  CI bakes in the first real numbers on its first green run, after
  which we promote those into the committed baseline. Diff your
  `results.json` against `baseline.json` to spot regressions.

`status` field semantics (per `BenchmarkResult.status`):

- `ok` — benchmark ran and met the migration plan's projection.
- `improved` — exceeded the projection (better than budgeted).
- `warn` — ran, but result is below projection or not yet observable
  (e.g. cache-token usage when the agent loop doesn't surface it).
  Often signals an unmerged feature dependency.
- `regressed` — measurably worse than the baseline; investigate.
- `skipped` — prerequisites missing (live env not enabled, baseline
  repo not checked out, optional module not present).

## Budgets

These are the targets the migration plan asks the harness to track.
Treat them as goals; the bench reports the actual delta and lets a
human decide if a regression is acceptable.

| Bench | Budget |
| --- | --- |
| `prefix-size` | 50–70% reduction in eager vs. lazy prefix tokens |
| `bundle-size` | ≥ 80% reduction vs. baseline tarball (stretch: 88%) |
| `tool-exec-time` | Median < 5 ms per tool on the fixture project |
| `cache-hits` | ~90% input-token cost reduction on warm turn |
| `first-token-latency` | Measurable reduction warm vs. cold |

The exact phrasing and rationale for these targets lives in the
migration plan; refresh this section if the plan moves.

## CI hookup

Not yet wired. The intent is for `.github/workflows/build.yml` (or a
new `bench.yml`) to run `pnpm bench` on push and PR with
`continue-on-error: true`, attach `results.json` as an artifact, and
post the markdown table as a PR comment. Live benchmarks are **never**
wired into CI; run them locally:

```sh
WIZARD_LIVE_BENCHMARK=1 pnpm bench --live
```

## Stubbed benches

Two benches currently emit `skipped` with a `TODO(phase-D-3)` marker
because they depend on a public agent-loop streaming entry point the
wizard does not yet expose:

- `cache-hits.bench.ts` — needs the AI SDK runner (or equivalent) to
  surface `cacheReadTokens` / `cacheWriteTokens` from `onFinish`.
- `first-token-latency.bench.ts` — needs a streaming surface that emits
  text deltas observable from the harness.

The stubs deliberately keep the row visible in `results.json` so the
schema stays stable and the wiring lands in one place when the
runtime is ready.

`tool-exec-time`'s `load_skill` row also degrades to `skipped` until a
public skill-loader module exists; the four fs-based tools always run.
